# 目标运营模型

Owner: `opl-health-platform`
Purpose: `hospital_operating_model`
State: `active_planning`
Machine boundary: 本文定义目标角色、责任与协作方式，不代表任何医院组织、权限、流程、系统或负责人已完成落地。

OPL Health Platform 的目标是在医院内形成一套可治理的医疗智能体运营体系。

它面向五类角色：医院管理者、科室负责人、医生和研究者、信息化团队、AI 团队。平台价值来自这些角色在同一套能力体系下协作，而不是单个智能体独立工作。

## 医院获得什么

医院获得的是一套医疗智能体平台能力：

- 把医学知识、临床规则、院内工具和专病流程组织成可复用能力。
- 让医生、研究者和科室在受控工作空间中使用医疗智能体。
- 让信息化团队管理数据接入、权限、部署和审计。
- 让 AI 团队开发、测试、发布和迭代医疗智能体。
- 让管理团队看到用量、风险、审查记录和交付结果。

## 角色分工

| 角色 | 使用平台做什么 | 主要关注点 |
| --- | --- | --- |
| 医院管理者 | 建设院级医疗 AI 能力、查看用量和风险 | 安全、审计、成本、责任边界 |
| 科室负责人 | 沉淀专病流程、科研流程和质控要求 | 专科价值、复用能力、审查质量 |
| 医生和研究者 | 在工作空间中完成科研、质控、随访和报告任务 | 专业准确性、效率、交付物 |
| 信息化团队 | 接入系统、配置权限、部署资源和审计策略 | 数据边界、系统稳定性、权限 |
| AI 团队 | 设计、测试、发布和改进医疗智能体 | 生命周期、测试、版本、运行反馈 |

## 平台运行方式

```text
医院场景定义
-> 医学知识与临床规则沉淀
-> 医疗工具与数据接入
-> 医疗智能体设计和测试
-> 管理端审批
-> 工作空间启用
-> 医生或研究者使用
-> 审查和交付记录
-> 版本改进
```

## 用户工作面

医生和研究者主要使用工作空间：

- 打开项目或专病任务。
- 选择已批准的医疗智能体。
- 提供资料、问题或任务目标。
- 查看任务进度、产物和审查反馈。
- 接收报告、表格、图表、病历摘要或质控结果。

管理者和信息化团队主要使用管理端：

- 管理组织、用户、科室和权限。
- 审批知识包、规则包、工具包和智能体版本。
- 配置工作空间、资源、连接器和审查策略。
- 查看用量、审计、风险和交付记录。

AI 团队主要使用构建流程：

- 把场景需求转成医疗智能体蓝图。
- 绑定 OPL Health Knowledge、OPL Health Protocol 和 OPL Health Tools。
- 设计测试案例和失败案例。
- 提交版本给管理端审批。
- 根据运行反馈改进智能体和能力包。

## 第一阶段目标

第一阶段先跑通一个科室或专病场景：

```text
一个明确场景
-> 一个知识包
-> 一个规则包
-> 一个工具包
-> 一个医疗智能体
-> 一个工作空间任务
-> 一份审查记录
-> 一个可交付结果
```

这条链路跑通后，再扩展更多专病、更多工具和医院级管理能力。
