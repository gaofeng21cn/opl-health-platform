# 专病质控助手

Owner: `opl-health-platform`
Purpose: `specialty_quality_scenario_reference`
State: `future_scenario`
Machine boundary: 本文是第二阶段候选场景参考，不证明指标、数据接入、质控流程、责任 owner 或运行能力已经就绪。

专病质控助手面向科室负责人、质控团队和医院管理团队，帮助他们围绕专病路径、指标和异常情况进行核查、解释和改进。

这是第二阶段更适合推进的场景，因为它需要更明确的数据接入、指标口径和责任边界。

## 用户

- 科室负责人。
- 质控医生。
- 医务管理部门。
- 数据和信息化团队。

## 典型输入

- 专病质控指标。
- 临床路径和科室制度。
- 病例摘要或结构化数据。
- 月度、季度或项目报表。
- 异常病例或异常指标清单。

## 需要的医学知识

- 专病诊疗指南。
- 医院和科室制度。
- 质控标准和管理口径。
- 指标解释材料。

## 需要的临床规则

- 指标定义。
- 数据纳入范围。
- 路径符合性规则。
- 异常判定规则。
- 人工复核和责任归属规则。

## 需要的工具

- 院内数据库或报表系统。
- EMR、HIS、LIS、PACS 的受控数据引用。
- 指标计算和图表工具。
- 质控报告生成工具。

## 智能体工作流

```text
质控目标定义
-> 指标和路径规则选择
-> 数据或病例引用
-> 指标核查
-> 异常解释
-> 改进建议草案
-> 人工复核
-> 质控报告交付
```

## 人工确认点

- 指标适用范围。
- 数据来源和统计口径。
- 异常解释。
- 改进建议。
- 对科室或医院发布的质控结论。

## 交付物

- 指标核查摘要。
- 异常病例或异常指标清单。
- 质控图表。
- 改进建议草案。
- 科室质控报告。
- 审查记录。

## 审查方式

OPL Health Review 重点检查：

- 指标定义和数据来源是否一致。
- 异常解释是否有证据支持。
- 改进建议是否保持在可执行范围内。
- 高风险结论是否经过人工确认。

## 与 OPL Cloud 的关系

- OPL Fabric 需要支持院内数据或报表系统的受控接入。
- OPL Console 需要管理科室权限和指标审批。
- OPL Ledger 需要记录数据来源、计算口径、图表和审查结果。
- OPL Workspace 展示指标、异常解释、报告草案和审查反馈。
